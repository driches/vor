# Eval Harness — Design

## Context

The Vor action currently runs as a single Sonnet 4.6 agent loop on every PR. A recent dogfood run cost ~$2.36 and hit the 40-turn cap with only 3 comments on a 63-file PR. Two related concerns:

1. **Cost-per-PR** is not sustainable for an always-on review on large repos. We want roughly the same review quality at a fraction of the cost.
2. **We can't tell which model strategy is cheaper without a fair comparison.** Anecdotal "Haiku felt fast" isn't a basis for production decisions.

This spec designs a **synthetic-bug eval harness** (Phase A) that runs the orchestrator against a fixed corpus of cases under multiple named model/config combinations, scores each run against planted ground-truth bugs, and renders a per-corpus comparison table. The harness's purpose is to let us measure cost-vs-recall tradeoffs before committing to a multi-stage architecture (Phase B, out of scope).

The user's chosen optimization goal: **same recall as Sonnet-only at ≥25% lower cost**. The user accepted Haiku-flags-with-Sonnet-validation-of-skips as the eventual Phase B architecture but explicitly wants data before committing to it.

## Out of scope (Phase B work)

- Multi-stage pipelines (Haiku triage → Sonnet review).
- Sample-based skip validation between stages.
- Per-stage cost telemetry inside a single run.
- Anything that requires a new `pipeline.stages[]` schema.

The harness in this spec MUST leave room for Phase B without pre-declaring its schema.

## Components

### 1. Pipeline configs (`configs/pipeline/*.yml`)

New top-level directory in this repo. Each YAML is a named config that the harness can invoke. Schema is a thin subset of the existing `ReviewConfig` shape (same Zod partial-merge as `.vor.yml`):

```yaml
# configs/pipeline/sonnet-only.yml — baseline, identical to current production default
model: claude-sonnet-4-6
max_turns: 40
severity:
  floor: minor
  max_comments_per_file: 5
  max_comments_total: 30
budget:
  max_input_tokens: 500000
  max_output_tokens: 50000
```

`opus-only.yml` and `haiku-only.yml` use the same shape, varying `model` and `max_turns`. Missing fields fall back to `DEFAULT_CONFIG`, so configs can be as short as `model: claude-haiku-4-5`.

**No `pipeline:` block in v1.** Configs that fail to parse against the v1 schema are skipped with a `logger.warn`. When Phase B lands, the schema gains a `pipeline.stages[]` block; v1 configs remain valid as single-stage configs.

### 2. Golden dataset (separate repo)

Lives outside this repo in a path configured via `GOLDEN_REPO_PATH` (env or CLI flag). Layout per case:

```
<GOLDEN_REPO>/cases/<id>/
  before/          # source snapshot — committed real code, often a small repo's HEAD
  plants.yml       # author-controlled list of mutations
  after/           # generated by scripts/plant.ts — snapshot with bugs planted
  truth.yml        # generated — TruthEntry per plant
  runs/            # one JSON per (timestamp, config), per §4
```

`before/` and `plants.yml` are authored; `after/` and `truth.yml` are derived from them by `scripts/plant.ts`. Both derived files are committed so reviewers can read the planted-bug diff in PRs to the golden corpus.

Plants apply **in `plants.yml` order**. Each template operates on the current state (i.e., previous plants are visible) and computes the truth entry against the file as it exists at apply-time. If two plants target the same file and the first shifts line numbers, the second's `line:` is interpreted against the post-first-plant content. Templates that insert (rather than replace) advance subsequent truth `line_range` values automatically.

### 3. Plant catalog (`scripts/plant/templates/*.ts`)

Each template is a small TypeScript function:

```ts
export interface PlantTemplate {
  type: string;                       // e.g. 'secret:aws-access-key'
  apply(source: string, config: PlantConfig): {
    mutated: string;
    truth: TruthEntry;
  };
}
```

v1 templates:

| Plant type | Tests detection by | What it does |
|---|---|---|
| `secret:aws-access-key` | secrets scanner | Insert `const k = "AKIAIOSFODNN7EXAMPLE";` |
| `secret:github-pat` | secrets scanner | Insert `ghp_xxx…40chars` |
| `secret:pem-private-key` | secrets scanner | Insert PEM header block |
| `sql-injection` | AI security reasoning | Insert `` db.query(`SELECT … ${input}`) `` |
| `path-traversal` | AI security reasoning | Insert `` fs.readFile(`./uploads/${path}`) `` |
| `eval-user-input` | AI security reasoning | Insert `eval(req.body.code)` |
| `vuln-dep:npm` | dependency-cve scanner | Add a known-vulnerable lockfile pin |
| `vuln-dep:pypi` | dependency-cve scanner | Add a CVE'd PyPI version pin |

Catalog grows as new bug shapes prove worth testing. Each template fully owns the file mutation and the resulting `TruthEntry` shape.

### 4. Truth entries (`truth.yml`)

```yaml
truths:
  - file: src/config/aws.ts
    line_range: [12, 12]
    bug_type: secret:aws-access-key
    severity: critical
    plant_id: 0
    category: vulnerability      # AI- or scanner-compatible category
```

`category` is the union of compatible categories that count as a "match" against this truth. Used by the scoring stage.

### 5. CLI (`scripts/eval.ts`, exposed via `npm run golden:eval`)

```
golden:eval --case <id>          --configs sonnet-only,opus-only,haiku-only
golden:eval --case <id>          --configs all
golden:eval --all                --configs sonnet-only            # corpus sweep on baseline
golden:eval --all                --configs all                    # full matrix
```

`--configs` is **required** — there's no default. Refusing to default prevents accidentally running a (potentially expensive) Anthropic API matrix on a misremembered command. `--configs all` expands to every `*.yml` under `configs/pipeline/` at invocation time.

Per `(case, config)` run:

1. Resolve the config (parse YAML, deep-merge over `DEFAULT_CONFIG`).
2. Invoke `runOrchestrator` against the case's `after/` snapshot. Anthropic API + Octokit are real; the action's existing diff/file/lockfile code reads from `after/`.
3. Capture the resulting `ReviewDraft` plus cost metrics (cache hits, turns, wall_ms, ended_reason).
4. Write `runs/<timestamp>-<config>.json`:

```ts
interface RunRecord {
  case_id: string;
  config_name: string;
  timestamp: string;
  config_resolved: ReviewConfig;
  cost: { input_tokens; output_tokens; cache_read; cache_creation; cost_usd; turns; wall_ms; ended_reason };
  findings: PostedComment[];     // the kept set after filter + dedup
}
```

### 6. Scoring (`scripts/score.ts`)

For each run record, compute per-case results:

```
for each truth T in case.truth.yml:
  finding F matches T iff:
    F.file_path === T.file
    AND |F.line - T.line_range.start| <= 3
    AND F.category is in T.category (the compatible set)
  if some F matches T → TP
  else                → FN

for each finding F:
  if no T matches F → FP (unaligned finding — noise but recorded)
```

Metrics per (case, config):

- `recall = TP / (TP + FN)`
- `precision = TP / (TP + FP)`
- `f1 = 2*P*R / (P + R)`
- `cost_per_tp = cost_usd / max(TP, 1)`
- `unaligned[]` — full list of findings that didn't match any truth, for promotion review

### 7. Report renderer (`scripts/render-report.ts`)

Two outputs per eval invocation:

**`<GOLDEN_REPO>/reports/<timestamp>-summary.md`** — corpus-level markdown table comparing each config against the baseline:

```markdown
| Case | Plants | sonnet-only | opus-only | haiku-only |
|---|---|---|---|---|
| nodeapi-pr-12 | 6 | R 100% / $2.30 (🟢 base) | R 100% / $4.10 (🟡 +78%) | R 67% / $0.18 (🔴 -33% R) |
| ... | ... | ... | ... | ... |
| **Aggregate** | 13 | R 92% / $5.05 | R 100% / $9.40 | R 81% / $0.38 |
```

Win cell colors per the user's chosen optimization goal ("same recall, lower cost"):
- 🟢 Win: `recall_challenger ≥ recall_baseline` AND `cost < baseline × 0.75`
- 🟡 Tradeoff: `recall_challenger ≥ recall_baseline` AND `cost < baseline`
- 🔴 Regression: `recall_challenger < recall_baseline`
- ⚪ Inconclusive: within ±5% on both axes

**`<GOLDEN_REPO>/cases/<id>/runs/<timestamp>-<config>-unaligned.md`** — per (case, config), the `unaligned[]` findings as a candidates-for-promotion table.

### 8. Promote command (`scripts/promote.ts`, exposed via `npm run golden:promote`)

```
golden:promote --case nodeapi-pr-12 --finding sonnet-only#3
```

Reads the most recent run JSON for that (case, config), copies finding `#3` into the case's `truth.yml` as a new entry, regenerates `truth.yml`. Most promotions don't require new planting (the finding was in the organic source); the comment in the truth entry records the originating config so we know who first surfaced it.

## Data flow

```
plants.yml ──┐
             ▼
        scripts/plant.ts ──▶ after/ + truth.yml
                                │
                                ▼
configs/pipeline/*.yml ──▶ scripts/eval.ts ──▶ runs/<ts>-<cfg>.json
                                                       │
                                                       ▼
                                               scripts/score.ts
                                                       │
                                                       ▼
                                          scripts/render-report.ts
                                                       │
                                                       ▼
                                         reports/<ts>-summary.md +
                                         per-case unaligned.md
```

## Phase B forward-compatibility

- Configs without a `pipeline:` block stay valid forever. Phase B adds the block; v1 single-stage configs continue working.
- `RunRecord.cost` is per-run today. Phase B adds `stages: [{name, model, cost: {...}}]` alongside; report renderer treats single-stage configs as a one-element stages array.
- Plant catalog grows with new bug shapes as Phase B reveals categories the harness needs to cover.

## Testing strategy

- **`scripts/plant.ts`** — unit tests per template. Each template gets a fixture `before/` and expected `after/` + `truth.yml`. Snapshot-style assertions.
- **`scripts/score.ts`** — unit tests over fabricated `(findings, truths)` pairs covering: exact match, 3-line slack hit, 4-line slack miss, category-compatibility hit, mismatched category miss, unaligned FP counting.
- **`scripts/eval.ts`** — one integration test that uses a fixture golden repo (committed under `tests/fixtures/golden-repo/`) with two tiny cases and runs a stubbed orchestrator (Anthropic + Octokit mocked at the module boundary, same pattern as `orchestrator.test.ts`). Asserts the run JSON shape + report output.

## Operational notes

- The golden corpus lives in a separate repo so it can grow independently and stay private if needed. Default path lookup: `~/code/vor-golden` or `$GOLDEN_REPO_PATH`.
- Each `golden:eval` run is real-money: it calls Anthropic. Eval should never run automatically on PRs to *this* repo; it's a manual command.
- The harness assumes `runOrchestrator` is callable with a synthetic input — no PR number, no GitHub API. v1 implementation will need a small adapter that fakes `prContext` from `after/` and routes `createReview` to a capture array instead of hitting GitHub. The existing `orchestrator.test.ts` already mocks all this; the adapter generalizes that mocking layer.

## First milestone after v1 ships

Pick 5 cases that represent the corpus (one small JS PR, one large Python PR, one infra repo with lockfile churn, one secrets-heavy migration, one cross-cutting refactor). Run all 3 single-model configs (`sonnet-only`, `opus-only`, `haiku-only`). Look at the report. If Haiku-only loses ≥1 case on subtle reasoning, that's the empirical foundation for Phase B's Haiku-triage + Sonnet-validate split.
